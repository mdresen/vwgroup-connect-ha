## Recommended v2.5.0 scope (~10h)

**Bundle as a single PR** to keep release-cadence simple:

1. **A1+A2+A3+A4 — daern field-coverage audit** (~5h)
   - Real-world fixture diff for a CUPRA/SEAT vehicle showing our `/v5/mycar` response vs daern's individual endpoints
   - If gaps found: add fallback parsers + new dataclass fields per the Scout Policy (parse → silence → entity)
   - Adds maybe 0-10 new diagnostic entities (depends on what's actually missing)

2. **C1 — ACTIVE_VENTILATION for Skoda ICE** (~1-2h)
   - Add capability to `_capabilities.py`
   - Add switch entity in `switch.py`
   - Mirror existing aux-heater switch pattern (no S-PIN needed for ventilation)
   - Service handler in `__init__.py` + i18n string

3. **C3 — Skoda Mk2 image-render audit** (~1.5h)
   - Single-vehicle real-world test against Christian (#75) Skoda Mk2 if we can ping him
   - If render-URLs come back empty: extend the parser to handle new shape

4. **M1+M2 — mitch-dc legacy documentation** (~2h)
   - Update `_community-projects.md` flagging mitch-dc as archived
   - New `docs/MIGRATING-FROM-MITCH-DC.md` guide (entity-ID mapping, automation rewrite hints, HACS swap-out)

**Total estimated**: 9-10 hours of focused work for a substantial v2.5.0 MINOR release.

## Alternative: split into v2.5.0 + v2.5.1

If 10h-in-one-PR feels too much, split:

- **v2.5.0**: daern field-coverage audit (#A1-A4) + ACTIVE_VENTILATION (C1) = ~7h
- **v2.5.1**: Skoda Mk2 image-renders (C3) + mitch-dc legacy docs (M1+M2) = ~3h

## What we DON'T act on (consciously)

- **mitch-dc archival → fork it**: tempting but no — daern is right that focused projects per brand age into maintenance mode. Our cross-brand approach is the better long-term play.
- **Adopting daern's addressable pattern**: HA coordinator already gives us this, just less explicit. Don't refactor for refactor's sake.
- **Adopting iOS User-Agent**: their `CUPRAApp%20-%20Store/20220503 CFNetwork/...` is dishonest about platform; our `OLACupra/2.15.0 (Android ...)` is more honest about being a HA-Linux user-agent.

## When to ship v2.5.0

No urgency — no open user-blocking bugs. Suggested cadence: pick this up when next batch of scout-reports or open issues drops, bundle the audit work alongside the user-driven fixes.

## How to act on it later

The audit docs in [`community-audits/`](community-audits/) are self-contained — anyone (you or future contributors) can pick a specific action (A1, C1, etc.) and ship it. The action plan above is just a suggested sequencing.
