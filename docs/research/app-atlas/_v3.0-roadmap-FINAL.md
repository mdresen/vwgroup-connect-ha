## 0. Strategic frame (locked)

### Three NEW constraints layered on top of master plan

1. **Push-first architecture** — wherever a brand supports push (FCM/MQTT/WebSocket), we use it as PRIMARY data source. Polling drops to BACKUP role (e.g. 30-60min cadence instead of 5-15min). Result: orders-of-magnitude fewer API calls + tokens + 12V wake-ups.
2. **Public-docs secrecy** — NEVER mention "extracted from APK" / "reverse-engineered" in any user-facing channel (README, CHANGELOG, release notes, GitHub Issues replies, blog posts, video transcripts). Use the public-language snippets in §6 below.
3. **Competitive moat** — structure the code so the VALUE LAYER is in higher-level orchestration (cadence tracker, capability gating, parser-health telemetry, multi-source consensus, scout policy) rather than just endpoint URLs. Make us hard to copy without doing the same accumulated work.

### Theme

"Suit Up: The Push Tech Edition" 🎩 (HIMYM Barney) with BBT sub-tracks per beta.

## 2. Push-first architecture (NEW primary constraint)

### Today's reality

Every brand polls every 5-15 minutes via HTTP REST. Even cars sitting in garage for hours consume:
- 12V wake-up energy → drains battery slowly on long parks
- API quota → eventually rate-limits or revokes
- OEM token-refresh storm → known cause of `invalid_credentials` cascades (smartmatic's #282 was triggered by this)

### v3.0 push-first model

```
                  ┌─────────────────────────┐
                  │   OEM Backend (Cariad/  │
                  │   OLA/mysmob/PPA/etc)   │
                  └────────┬────────────────┘
                           │ Vehicle state change
                           ▼
                  ┌─────────────────────────┐
                  │   FCM / MQTT / WS push  │
                  └────────┬────────────────┘
                           │ data-message
                           ▼
              ┌─────────────────────────────────┐
              │ vag_connect cariad/push/_base   │
              │ → normalize to VagPushEvent     │
              └────────┬────────────────────────┘
                       │ event.type
                       ▼
              ┌─────────────────────────────────┐
              │ Coordinator bridge:             │
              │ - OPERATION_FEEDBACK → entity   │
              │   refresh via                   │
              │   async_set_updated_data        │
              │ - SERVICE_EVENT → ring-buffer   │
              │ - VEHICLE_EVENT → log only      │
              └─────────────────────────────────┘
```

### Polling cadence after push is healthy

| Push state for VIN | Fast-coord cadence | Slow-coord cadence | API saved |
|---|---|---|---|
| Push channel DOWN (fallback) | 5min (today) | 4h | 0% |
| Push channel healthy < 1h (warm-up) | 5min | 4h | 0% |
| Push channel healthy 1-6h | 15min | 6h | ~66% |
| Push channel healthy > 6h (steady-state) | 30min | 24h | ~83% |
| Push channel DOWN > 30min (recovery probe) | 5min | 4h | 0% |

Net result: **most users save 80%+ API quota** + see faster state updates than today. Win-win.

## 4. Competitive moat strategy

### What's hard to copy

Even if a competitor reads all our public-shipped code, they can't easily replicate:

1. **User-tester pool** — smartmatic, matthias, goncal, Gerhard, christianmhz, rocksandclouds, whaak58, Daniel-Walter, tritanium73, KimmoT727, Brinki99, arvcer, j4x5mgq94b — accumulated through 18 months of community engagement.
2. **App Atlas daily monitoring infrastructure** — they'd need to clone the workflow + maintain it + accumulate per-version history.
3. **4-layer OLA defense-in-depth** — copy-pasteable, but the OPTIONSFLOW + REPAIR CARD + MULTI-VERSION FALLBACK orchestration is the value.
4. **Scout Policy + T1-T5 tier system** — copy-pasteable, but the 67-path classified catalog reflects accumulated debugging knowledge.
5. **Per-VIN capability gating (Phase 3)** — they'd need their own capability discovery + cache.
6. **Multi-source consensus polling** (3+ sources before opening PR) — they'd need to maintain their own watcher.
7. **Cumulative bug-fix knowledge** — which fields work on which vehicle variants, which silencers are T1 vs T5, etc.
8. **Parser-health telemetry** (β1) — original architectural pattern we're shipping FIRST.
9. **Cadence-tracker module** (β1) — same.
10. **Push abstraction with token-rotation** (β3) — myskoda's #1091 issue is still open; we're shipping the fix first.

### What's easy to copy (don't bother protecting)

- Endpoint URLs — anyone with apktool can extract these in 10 minutes.
- Firebase project IDs / sender IDs — same.
- Basic entity adds — once we ship the entity-IDs, anyone can read them.

### Moat positioning

**"We're not a fork. We're the orchestration layer."** The endpoint paths are commodities; the value is in HOW we use them — cadence, fallback, gating, drift-detection, push-bridge, capability-aware feature gating.

### Anti-copy tactics

1. **Don't blog about HOW we discovered things** — share WHAT users get.
2. **Don't open-source the App Atlas APK download/extract pipeline as a standalone** — keep it in our repo only.
3. **Release docs phrase things from user-benefit POV** — "your car wakes up less" not "we save 83% API calls".
4. **Architecture decision records (ADRs)** stay in `docs/internal/` if we create them.
5. **Bruno test collections** with discovered endpoints — keep current pattern of `.local.bru` gitignored; share `.bru` templates but not specific creds.

## 5. Risk register (additions for v3.0)

| Risk | Mitigation |
|---|---|
| FCM registration mimics OEM app → legal gray area | We register with FCM using PUBLIC sender IDs (extractable by anyone, not secret). | User shares `.app-atlas-apk-cache/{brand}.json` containing brand Firebase IDs in a public issue | Add `_scrub_pii.py` rule: detect Firebase project IDs + warn user. Diagnostic dumps already scrub VINs/tokens/GPS — extend to Firebase IDs. |
