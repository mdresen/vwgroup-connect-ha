# Cross-Brand App Atlas

> Daily-refreshed knowledge base of what each VAG-brand Android app
> actually talks to. Generated by
> [`scripts/app_atlas/build_atlas.py`](../../../scripts/app_atlas/build_atlas.py)
> via [`.github/workflows/app-atlas-builder.yml`](../../../.github/workflows/app-atlas-builder.yml).

## Why this exists

Every reverse-engineering project today is **brand-siloed**:
- CarConnectivity has separate connectors per brand
- PyCupra only knows CUPRA
- mitch-dc only knows VW EU
- volkswagencarnet covers VW + Audi but is legacy
- audi_connect_ha only knows Audi

Nobody systematically asks **"does the Audi app talk to OLA?"** or
**"is VW silently consolidating backends?"** This atlas fills that
gap by walking all 7 brand apps daily and tracking what they do.

## How discoveries flow into the integration

When the atlas surfaces something actionable (e.g. an Audi APK
references OLA endpoints), the discovery triggers a separate PR
following the regular contribution workflow:

1. Atlas detects new endpoint / header / endpoint shift
2. Maintainer reviews — is this actionable for our integration?
3. If yes: PR adds parser / backend route / header injection
4. Atlas page updates to reflect that we now use the discovery

## Daily run

The workflow runs every day at 04:00 UTC. When versions change,
it opens an auto-PR for human review (no auto-merge — same safety
pattern as the upstream OLA watcher).

## Manual run

```bash
# Run locally (no network mocks — actually scrapes APKMirror):
python scripts/app_atlas/build_atlas.py

# Single brand for debugging:
python scripts/app_atlas/build_atlas.py --brand audi

# Dry run (show what would change, don't write):
python scripts/app_atlas/build_atlas.py --dry-run
```
