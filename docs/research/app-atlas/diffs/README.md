# App Atlas — Deep-Diff Reports (Phase A.3)

This directory holds **per-brand × per-version-pair** semantic diff
reports generated by
[`.github/workflows/app-atlas-deep-diff.yml`](../../../../.github/workflows/app-atlas-deep-diff.yml).

## When to generate a report

Run the workflow manually when:

1. The daily atlas watcher reports a **major version bump** for a
   brand you care about (e.g. CUPRA 2.16 → 3.0)
2. You're investigating a suspected backend change (new endpoint,
   new header requirement) and want ground-truth from the actual app
3. You're preparing a v2.X release and want to proactively check if
   the apps have evolved their auth flow or endpoint set

## How to generate a report

1. Go to the [Actions tab](../../../../actions) → "App Atlas Deep
   Diff (manual)"
2. Click "Run workflow"
3. Pick:
   - Brand (one of 7)
   - Old version-name (e.g. `2.15.0`)
   - New version-name (e.g. `2.16.0`)
4. Wait ~15-30 min (jadx full decompile of 2 APKs)
5. Workflow auto-opens a PR with the generated report
6. Review + merge (or close if no actionable findings)

## What's in each report

| Section | Content |
|---|---|
| Summary | Counts of added/removed URLs, headers, scopes |
| URLs added/removed | The highest-signal change — new backend endpoints the app started talking to |
| Header keys added/removed | Potential future enforcement signal — pre-emptively add to our outgoing requests |
| OAuth scopes added/removed | Auth-flow evolution — review our IDP scope chain |
| Action items | Triage hints calling out OLA-host references, `app-*` headers, scope changes |

## What's NOT in the reports

- Raw decompiled source (too big, ~100MB+ per APK)
- UI/layout changes (we don't care)
- Generated-code changes (obfuscator renames create false-positive noise; we filter those)
- Permission changes (those would come from AndroidManifest analysis — could be added as Phase A.3.1 if useful)

## Limitations (current implementation)

- APKCombo only serves the latest version per app. Per-version
  fetching requires Phase A.3.1 enhancement (navigate to
  per-version pages instead of `/download/apk`).
- Until A.3.1, the workflow effectively compares "latest" to "latest"
  if you run it for the same version twice. Useful for checking
  baseline state but doesn't yet enable historical "look back".
